````md
# wechat-md-server 产品需求文档（PRD）

> 版本：V5.2  
> 最后更新：2026-05-05  
> 项目定位：面向 Obsidian 用户的自托管信息入口与知识入库服务

---

## 1. 产品定位

### 1.1 一句话定义

面向 Obsidian 用户的自托管本地服务，将微信公众号文章和普通网页长文自动抓取、清洗、转换为结构化 Markdown，并同步到 Obsidian 笔记库。

---

### 1.2 目标用户

| 用户画像             | 核心诉求                                            |
| -------------------- | --------------------------------------------------- |
| Obsidian 重度用户    | 看到好文章想快速入库，建立个人知识库                |
| 信息工作者           | 需要批量归档公众号文章用于后续检索和整理            |
| 知识管理爱好者       | 希望文章入库时自带 AI 生成的摘要、标签、Frontmatter |
| 团队/小型组织        | 通过 Bot 接口共享文章转换能力                       |
| NAS / 本地自托管用户 | 希望服务运行在本地设备，不强依赖公网 VPS            |
| 创作者 / 深度学习者  | 希望把外部文章沉淀为长期可用的资料库和创作素材库    |

---

### 1.3 解决的核心问题

1. **公众号文章难以长期保存**：微信文章 URL 会失效、阅读体验受限于微信客户端。
2. **手动复制粘贴效率低**：格式丢失、图片无法保留、批量操作不现实。
3. **Obsidian 入库流程断裂**：转换后的 Markdown 需要手动拖入 Vault，无法形成自动化闭环。
4. **多入口收录缺少统一处理链路**：Web、Bot、批量、自动同步等入口需要统一进入同一套转换 Pipeline。
5. **本地部署受公网限制**：Bot Webhook 默认依赖公网地址，不利于家庭 NAS 或本地服务器部署。
6. **主题内容发现能力不足**：用户目前需要先手动找到文章链接，缺少按主题搜索公众号文章并批量入库的入口。

---

### 1.4 产品边界

#### 做

- 微信公众号文章 → Markdown → Obsidian 同步
- 普通网页文章 → Markdown → Obsidian 同步
- 单篇 / 批量 URL 转换
- Bot 入口触发
- 公众号自动同步源
- 图链处理
- AI 摘要 / 标签 / Frontmatter
- FNS / Obsidian 同步
- Docker / VPS / NAS / 本地部署
- 主题搜索微信公众号文章候选列表，并由用户勾选入库

#### 不做

- 不是通用爬虫平台
- 不做视频 / 音频转换
- 不做知识图谱
- 不做笔记编辑器
- 不做 SaaS 托管服务
- 不承诺精准完整的全网微信公众号搜索
- 不默认绕过平台访问限制
- 不做大规模自动采集

---

#### 3.2.3 转换 Pipeline

```text
URL 输入
→ 来源识别
→ HTML 抓取（带缓存）
→ HTML 清洗
→ 图片处理
→ 正文提取
→ HTML → Markdown 转换
→ 后处理（格式规范化）
→ AI 润色（可选）
→ 输出 Markdown 文件
→ 同步到 Obsidian
```
````

---

#### 3.2.4 Pipeline 环节要求

| 环节          | 要求                                                                                           |
| ------------- | ---------------------------------------------------------------------------------------------- |
| 来源识别      | 根据 URL 域名自动判断 wechat / web / zhihu                                                     |
| HTML 抓取     | 伪装浏览器 User-Agent，支持磁盘缓存（SHA-256 索引）                                            |
| HTML 清洗     | 移除 script/style/comment，解析 lazy-load 图片（data-src → src），移除推广/引流/加群等噪音内容 |
| 图片处理      | 支持微信原链和 S3 图床外链                                                                     |
| Markdown 转换 | 自定义 HTMLParser，支持标题/列表/表格/代码块/引用/行内格式                                     |
| 后处理        | 空行规范化、代码块修复、表格分隔符补全                                                         |
| 输出          | 自动编号目录结构（001_Title/、002_Title/...）                                                  |

### 4.2 安全

| 要求             | 实现方式                                                   |
| ---------------- | ---------------------------------------------------------- |
| 密码安全         | PBKDF2-SHA256，200,000 次迭代                              |
| 会话安全         | HMAC 签名 Token + CSRF Double Cookie                       |
| 敏感数据加密     | Fernet 对称加密（密钥由环境变量注入）                      |
| 登录防暴力破解   | 10 分钟内 5 次失败锁定 15 分钟                             |
| 安全响应头       | X-Frame-Options / X-Content-Type-Options / Referrer-Policy |
| 生产 Cookie      | SESSION_COOKIE_SECURE=true                                 |
| Bot 白名单       | Telegram / 飞书可限制允许使用的用户 ID                     |
| NAS 模式默认安全 | Web 管理页默认仅建议局域网访问                             |

| 任务可追溯         | 每次转换生成 execution 记录，记录触发渠道、状态、AI 使用情况    |

## 11. 追加功能：本地 / NAS 部署与 Bot Polling 入口

---

### 11.1 功能背景

当前 Telegram Bot 和飞书 Bot 默认依赖 webhook 接收消息，因此服务通常需要部署在 VPS 或具备公网访问能力的服务器上。

为了支持用户将本项目部署在本地服务器、家庭 NAS 或无公网 IP 环境中，需要新增 **Bot 主动接收模式**：

- Telegram 支持 Polling 模式
- 飞书支持长连接模式
- Web 管理页可仅在局域网访问
- FNS 可与本项目同机部署，也可部署在其他可访问机器上

该功能用于降低自托管部署门槛，使用户不必强制购买 VPS 或配置公网域名。

---

### 11.2 一句话定义

允许用户在无公网 IP 的本地服务器或 NAS 上运行本项目，并通过 Telegram Polling / 飞书长连接接收文章链接，触发文章转换与 Obsidian 同步。

---

### 11.3 新增使用场景

| #     | 角色           | 场景                                              | 预期结果                                      |
| ----- | -------------- | ------------------------------------------------- | --------------------------------------------- |
| US-10 | NAS 用户       | 用户将本项目部署在家庭 NAS 上，没有公网 IP        | Telegram / 飞书入口仍可正常接收链接并触发转换 |
| US-11 | 本地自托管用户 | 用户只希望 Web 管理页在局域网访问                 | 无需暴露公网，Bot 入口仍可远程提交文章链接    |
| US-12 | 隐私敏感用户   | 用户希望文章数据、缓存、SQLite 数据库都保存在本地 | 服务和 FNS 均可部署在 NAS，本地完成入库同步   |
| US-13 | 非技术用户     | 用户不想配置域名、反向代理、Webhook 回调          | 使用 Polling / 长连接模式完成 Bot 接入        |

---

### 11.4 功能范围

#### 11.4.1 Telegram 接收模式

| 模式      | 描述                               | 适用场景                    |
| --------- | ---------------------------------- | --------------------------- |
| `webhook` | Telegram 将消息推送到公网 URL      | VPS / 公网部署              |
| `polling` | 服务主动调用 Telegram API 拉取消息 | 本地 / NAS / 无公网 IP 部署 |

#### Telegram Polling 要求

- 服务启动后可自动启动 Telegram polling worker
- polling worker 定时拉取 Telegram 更新消息
- polling 模式不要求配置公网 Webhook URL
- polling 模式下仍复用现有 Telegram 消息解析逻辑
- polling 拉取到的消息必须进入统一任务系统
- polling 与 webhook 二选一，不同时启用
- 切换到 polling 模式时，应提示用户清理旧 webhook
- 任务记录中需要保存 `trigger_channel=telegram` 和 `receive_mode=polling`

#### Telegram 新增配置项

| 配置项             | 描述                                  | 默认值    |
| ------------------ | ------------------------------------- | --------- |
| `TG_ENABLED`       | 是否启用 Telegram Bot                 | `false`   |
| `TG_BOT_TOKEN`     | Telegram Bot Token                    | 空        |
| `TG_RECEIVE_MODE`  | 接收模式：`webhook` / `polling`       | `webhook` |
| `TG_WEBHOOK_URL`   | webhook 模式下的公网回调地址          | 空        |
| `TG_POLL_INTERVAL` | polling 间隔秒数                      | `2`       |
| `TG_ALLOWED_USERS` | 允许使用 Bot 的 Telegram 用户 ID 列表 | 空        |

---

#### 11.4.2 飞书接收模式

| 模式              | 描述                             | 适用场景                    |
| ----------------- | -------------------------------- | --------------------------- |
| `webhook`         | 飞书事件订阅将消息推送到公网 URL | VPS / 公网部署              |
| `long_connection` | 服务主动建立飞书长连接接收事件   | 本地 / NAS / 无公网 IP 部署 |

#### 飞书长连接要求

- 服务启动后可自动启动飞书长连接 worker
- 长连接模式不要求配置公网事件回调 URL

- 断线、重连、鉴权失败需要记录日志
- 长连接收到的消息必须进入统一任务系统
- 长连接模式复用现有飞书消息解析逻辑
- webhook 与 long_connection 二选一，不同时启用
- 任务记录中需要保存 `trigger_channel=feishu` 和 `receive_mode=long_connection`

| `FEISHU_WEBHOOK_PATH`  | webhook 模式下的回调路径                | `/api/bot/feishu/webhook` |
| `FEISHU_ALLOWED_USERS` | 允许使用 Bot 的飞书用户 ID 列表         | 空                        |

### 11.5 统一 Bot 消息处理

无论消息来自：

- Telegram Webhook
- Telegram Polling
- 飞书 Webhook
- 飞书长连接
- 统一 Webhook 入口

都必须先转换为统一内部消息结构：

```json
{
  "trigger_channel": "telegram",
  "receive_mode": "polling",
  "sender_id": "123456",
  "chat_id": "123456",
  "message_id": "987654",
  "raw_text": "https://mp.weixin.qq.com/s/xxxx",
  "urls": ["https://mp.weixin.qq.com/s/xxxx"],
  "created_at": "2026-05-05T00:00:00Z"
}
```

#### 统一处理规则

- Bot 入口只负责接收消息、解析链接、创建任务
- Bot 入口不直接执行文章转换
- 文章转换必须进入统一任务系统
- 所有 Bot 触发任务必须记录：
  - `trigger_channel`
  - `receive_mode`
  - `sender_id`
  - `message_id`

- Bot 回执基于任务状态返回
- Webhook / Polling / 长连接模式必须共用同一套任务创建逻辑

## 12. 追加功能：主题搜索入库

---

### 12.1 功能背景

当前系统主要支持用户通过 URL、Bot、公众号同步源等方式提交文章。

但在真实使用中，用户经常不是已经拥有某篇文章链接，而是想围绕某个主题寻找微信公众号文章。例如：

- 想找“AI 编程工作流”相关公众号文章
- 想找“Obsidian 知识管理”相关文章
- 想找某个产品、工具、技术主题的优质文章
- 想做选题资料收集

微信官方不提供公开可用的全局文章搜索 API，因此本功能不承诺“精准全网搜索”，而是提供一个 **主题文章发现入口**：

> 输入主题关键词，搜索公开微信公众号文章候选结果，用户人工勾选后批量入库。

---

### 12.2 一句话定义

用户输入一个主题关键词，系统搜索公开微信公众号相关文章，返回候选列表，用户勾选后批量进入现有文章转换与 Obsidian 入库 Pipeline。

---

### 12.3 功能定位

该功能不是：

- 不是官方微信公众号全局搜索 API
- 不是精准全网搜索引擎
- 不是自动全量采集工具
- 不是替代微信“搜一搜”

该功能是：

- 主题文章发现页
- 公众号文章候选搜索入口
- 人工筛选后批量入库工具
- 服务于 Obsidian 资料库建设的内容发现模块

---

### 12.4 使用流程

```text
输入主题关键词
→ 搜索公开微信公众号文章
→ 返回前 10~20 条候选文章
→ 用户查看标题、公众号、摘要、时间
→ 用户勾选需要入库的文章
→ 批量创建转换任务
→ 进入现有 Pipeline
→ 图链处理 / AI 摘要 / Frontmatter
→ 同步到 Obsidian
```

| 功能         | 描述                                       | 优先级 |
| ------------ | ------------------------------------------ | ------ |

### 12.11 与现有 Pipeline 的关系

主题搜索功能只负责：

- 搜索候选文章
- 展示候选列表
- 接收用户勾选
- 创建批量入库任务

不负责：

- 文章抓取
- 图片处理
- Markdown 转换
- AI 润色
- FNS 同步

勾选后必须进入现有统一 Pipeline：

```text
SearchResult.url
→ 批量入库任务
→ 来源识别
→ HTML 抓取
→ 图片处理
→ Markdown 转换
→ AI 润色
→ FNS 同步
```

### 12.12 非功能要求

| 要求         | 说明                                           |
| ------------ | ---------------------------------------------- |
| 默认限量     | 默认返回 10 条结果，避免过度请求               |
| 请求限频     | 同一用户短时间内限制搜索频率                   |
| 搜索缓存     | 同一关键词短时间内优先使用缓存                 |
| 失败可解释   | 搜索失败时显示具体原因，而不是空白             |
| 不阻塞主服务 | 搜索请求不得影响文章转换任务                   |
| 候选人工确认 | 搜索结果必须由用户勾选后才入库，不自动全量抓取 |

### 12.13 约束与风险

| 类别          | 描述                                               |
| ------------- | -------------------------------------------------- |
| 官方 API 缺失 | 微信官方不提供公开全局文章搜索 API                 |
| 搜索完整性    | 搜索结果不保证覆盖所有相关文章                     |
| 结果差异      | 搜索结果可能与微信客户端“搜一搜”不同               |
| 访问限制      | 第三方搜索入口可能出现验证码、限流、封禁           |
| 合规边界      | 本功能只作为公开文章发现入口，不用于大规模自动采集 |
| 结果质量      | 返回文章可能存在标题相关但内容不相关的问题         |
| 维护成本      | 搜索 Provider 页面结构变化可能导致失效             |

### 12.14 MVP 实现顺序

#### 第 1 步：单 Provider 搜索

- 新增 `/search` 页面
- 支持输入关键词
- 接入一个搜索 Provider
- 返回前 10 条候选结果

#### 第 2 步：候选列表勾选入库

- 列表展示标题、公众号、摘要、时间、链接
- 支持多选
- 勾选后调用现有批量入库 API

#### 第 3 步：搜索缓存

- 保存搜索关键词和结果
- 短时间内重复搜索优先返回缓存

#### 第 4 步：已入库标记

- 根据 URL / 标题匹配已有文章
- 搜索列表显示“已入库”

#### 第 5 步：多 Provider 扩展

- 支持 `sogou_weixin`
- 支持 `web_search`
- 支持 `local_cache`

### 12.15 推荐实现边界

MVP 阶段只做：

```text
主题关键词
→ 返回候选文章
→ 用户勾选
→ 批量入库
```

暂不做：

- 自动全量抓取
- 大规模分页搜索
- 搜索结果智能排序
- 搜索结果 AI 预分析
- 复杂多搜索源合并去重
- 类似微信“搜一搜”的完整体验

主题搜索入库不是“全局精准搜索”，而是一个 **微信公众号文章发现入口**。

它的目标是帮助用户围绕主题快速找到候选文章，再通过人工筛选批量进入现有 Obsidian 入库流程。
